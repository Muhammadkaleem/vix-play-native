# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

VixPlay is a native Android media player ŌĆö a modern alternative to MX Player. Kotlin + Jetpack Compose, Media3/ExoPlayer + FFmpeg extension, indigoŌåÆviolet dark-first design. The project is currently an empty Android scaffold; no source files exist yet beyond placeholder tests.

Full product docs are in `vix-play-docs/media-player-docs/`. Design assets (color tokens, typography, icons, launcher icon) are in `config-vix-play/player-design-assets/`.

## Architectural decisions

These were locked in during design review ŌĆö don't revisit without a specific reason:

| Concern | Decision |
|---|---|
| **Module structure** | Single-module for now. Extract to multi-module after MVP compiles and plays a video. The tech-architecture doc shows the target multi-module layout for reference only. |
| **DI** | Hilt. `PlayerController` scoped `@ActivityRetainedScoped` ŌĆö survives rotation, torn down on back. Migrate to `MediaSessionService` when background audio lands (Roadmap Step 5). |
| **Navigation** | Navigation Compose with `@Serializable` destination classes (2.8+). `ACTION_VIEW` external intents map to deep-link URIs. |
| **Application ID** | `com.devbytes.vixplayer.app` ŌĆö this is permanent once the Play Store listing is created. |
| **Storage** | MediaStore + granular media permissions as primary; SAF tree access for folder browsing; `MANAGE_EXTERNAL_STORAGE` as opt-in power mode only (with Play Console declaration). |
| **FFmpeg** | Media3 FFmpeg extension built via Google's provided script, checked in as a local `.aar`. No custom FFmpeg build. |
| **AC3/DTS** | Omitted. HW decode covers most devices. Show the codec name in the unsupported-format error; revisit licensing at 1.0. |
| **Subtitles** | Media3 built-in renderer (SRT + basic SSA) for MVP. libass (full styled ASS/SSA) is a fast-follow once playback is stable ŌĆö not a blocker. |
| **Pro entitlement** | Local Play verification via Google Play Billing v6. No backend verification until a subscription tier exists. |
| **Analytics** | Firebase Crashlytics only. No Firebase Analytics for MVP ŌĆö add it at beta when real users exist. `google-services.json` goes in `app/`. |
| **Ads** | AdMob. Banner only in library chrome; never on the player surface. No interstitials during playback. |
| **Theming** | Single `PlayerTheme` composable with an internal `Build.VERSION.SDK_INT >= 31` branch for dynamic color. No hardcoded colors anywhere ŌĆö every component reads semantic tokens from `Tokens.kt`. |
| **Android TV** | Compose for TV (`androidx.tv`), not Leanback. Keeps the entire UI layer in one paradigm. |
| **Private folder recovery** | Recovery code shown at PIN setup (hashed locally). No email recovery ŌĆö that requires a backend. |

## Key constraints

- **No hardcoded colors.** Every component reads semantic tokens. This is the #1 theming rule ŌĆö retrofitting is the worst rewrite.
- **No ads on the player surface.** AdMob banners only in library chrome.
- **AC3/DTS omitted** ŌĆö do not add software decoders without legal review of the Dolby/DTS license situation.
- **FFmpeg must be LGPL build** ŌĆö audit build flags before adding any GPL-only components; GPL taints the whole app.
- `minSdk = 24`, `targetSdk = 36`. Material You (`dynamicColorScheme`) requires runtime API 31+ check; `PlayerTheme` handles this internally.
- Resume positions use MMKV (not DataStore) ŌĆö DataStore's coroutine overhead is too slow for per-seek writes.

### 2026-07-23 ŌĆö Trash instead of delete (API 30+) ŌĆö DONE, **not yet device-verified**
- **Directly de-risks the previous entry.** Bulk delete shipped irreversible and unverified, which was the standing concern with building it before a device pass. `MediaStore.createTrashRequest` makes the same operation **recoverable for ~30 days** on API 30+, so the riskiest feature in the app becomes forgiving. Taken ahead of its queue position for that reason.
- **Q1=A ŌĆö trash replaces delete; no permanent-delete action remains on API 30+.** `createTrashRequest(resolver, uris, true)` shows the **same OS confirmation** `createDeleteRequest` did, so nothing about the confirmation model changed. Rejected B (offer both ŌĆö putting an irreversible action one tap from a reversible one with near-identical labels is exactly the pairing that causes accidents) and C (keep permanent delete ŌĆö leaves the app's one irreversible operation irreversible when the platform offers recovery for free).
- **The real trade, named not hidden:** on API 30+ the user can **no longer permanently delete from inside VixPlay**. That's a genuine reduction in capability, accepted because permanent removal belongs to the system trash UI, which already exists and is where Android users are trained to look.
- **Below API 30 there is no trash concept**, so those devices keep the permanent delete and the app's own confirmation dialog, whose wording stays explicitly unambiguous about permanence. **Labels differ by API level on purpose** ŌĆö `deleteIsRecoverable` drives "Move to trash" vs "Delete", and the result message says "moved to trash" vs "deleted", so the UI promises exactly what *that* device will do rather than a generic phrase that's wrong on half of them.
- **Q2=A ŌĆö no in-app trash view; rely on the system.** Trashed media appears in the device's Files/Photos apps, which already have restore UI. **The honest cost is discoverability** ŌĆö a user who wants something back must know to look in another app. Rejected B (an in-app Trash screen ŌĆö genuinely better UX, and logged as the natural follow-up, but it's a new screen plus a new query mode and, critically, it would reintroduce a permanent-delete action, undoing what Q1 just removed) and C (grey trashed items in the library ŌĆö clutters the list with things the user just asked to remove).
- **The re-query and count logic needed no changes at all**: default MediaStore queries exclude trashed items, so a trashed track disappears from the library exactly as a deleted one did, and the "what actually happened" diff keeps working untouched. The queue reconciliation likewise still applies ŌĆö a trashed track is gone from the library either way.
- **Implementation snag worth recording:** the first attempt to add `isRecoverable()` silently no-op'd because the `python` replace anchored on a KDoc that had already been rewritten earlier in the same pass. It surfaced as an unresolved reference at compile ŌĆö a reminder that string-anchored edits fail *silently* when the anchor drifts, unlike a failed patch.
- Files: `MediaDeleter.kt` (`createDeleteRequest` ŌåÆ `createTrashRequest`, +`isRecoverable()`, KDoc rewritten for the trash model); `AudioLibraryViewModel.kt` (+`deleteIsRecoverable`, verb-aware result message); `AudioLibraryScreen.kt` (API-aware action label, dialog comment). Compiles clean; `assembleDebug` OK.
- **Deferred:** an in-app Trash screen with restore (`QUERY_ARG_MATCH_TRASHED`), trash for video/folder browser once multi-select is lifted there, and surfacing how long items remain recoverable (the retention window is the system's, not ours to state precisely).

### 2026-07-23 ŌĆö Bulk delete (multi-select) ŌĆö DONE, **not yet device-verified. Destructive ŌĆö verify before trusting.**
- **Built against a stated recommendation.** I twice advised doing the device pass *first*, because every other unverified feature fails recoverably (a wrong queue, a blank notification, a janky drag) while this one destroys the user's files. The user reaffirmed, so it was built ŌĆö with the safety design tightened accordingly. **This is the one feature where an unverified selection bug is not recoverable.**
- **Q1=A ŌĆö use the platform's confirmation where it exists; supply our own only where it doesn't.** New `MediaDeleter` handles all three Android flows: **API 30+** `MediaStore.createDeleteRequest` ŌåÆ the **OS** shows its own dialog with thumbnails; **API 29** direct delete throws `RecoverableSecurityException` whose `IntentSender` yields the same system prompt; **API 24ŌĆō28** direct delete under the `WRITE_EXTERNAL_STORAGE` the manifest already caps at `maxSdkVersion="28"` ŌĆö **no system confirmation exists on that path**, so the app shows an `AlertDialog` naming the count and stating it can't be undone. Rejected B (always our dialog *plus* the system one ŌĆö two dialogs for one action trains people to dismiss without reading, making confirmation *less* effective) and C (30+ only ŌĆö `minSdk` is 24, so it would silently do nothing for a slice of users).
- **Safety hole found and closed during implementation.** `systemConfirms()` initially returned `SDK_INT >= Q` (29). But on 29 the system prompt appears **only for files the app doesn't own** ŌĆö an app-owned file would have been deleted **silently, with no confirmation from anyone**. In practice the audio library holds the user's own music so it would always prompt, but "in practice" is not a safety argument for an irreversible operation. Changed to `>= R` (30). Consequence: on API 29 a non-owned file may show **two** prompts (ours, then the system's). That redundancy is the correct trade ŌĆö **no path may delete without at least one confirmation**.
- **Q2=A ŌĆö re-query MediaStore as the source of truth, then reconcile the queue.** After the result returns: reload the library, clear the selection, close any drill, and report what *actually* happened by diffing the track count before and after ŌĆö not by reporting what was requested. **The user can deselect items inside the system dialog**, so the request is not the outcome. Rejected B (optimistically drop rows ŌĆö if deletion partially failed the list would claim files are gone while they're still on disk, the worst failure mode a delete feature can have) and C (skip queue reconciliation ŌĆö deleting the playing track would leave ExoPlayer on a dead URI and surface as a playback error, reading as a crash).
- **`PlayerController.removeFromQueue(uris)`** walks the queue backwards (so removals don't shift indices still to be checked), drops matches, and stops + clears + sets `PlaybackKind.NONE` if the queue empties. Deleting the current track therefore skips onward rather than erroring.
- **Playlists needed no work**: `rowsFor()` already resolves each item against the live library, so entries pointing at deleted files automatically become "Unavailable ŌĆö file missing" and are excluded from playback ŌĆö exactly the behaviour the playlists pass built for this case.
- **Outcome reported honestly**: "3 files deleted" / "1 file deleted" / "Nothing was deleted" / "Delete cancelled", from the observed count difference.
- Files: `+data/repository/MediaDeleter.kt` (`DeleteResult`); `PlayerController.kt` (+`removeFromQueue`); `AudioLibraryViewModel.kt` (+`deleteSelection`, `onDeleteConsentResult`, `finishDelete`, one-shot `message`); `AudioLibraryScreen.kt` (delete action, `StartIntentSenderForResult` launcher, API 24ŌĆō28 confirm dialog, toast). Compiles clean; `assembleDebug` OK.
- **Deferred:** undo (would need a trash/staging area ŌĆö MediaStore deletion is immediate), delete from the video library and folder browser, and `MediaStore.createTrashRequest` (API 30+ soft-delete, a better model than permanent deletion and worth revisiting).
