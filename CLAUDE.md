# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

VixPlay is a native Android media player — a modern alternative to MX Player. Kotlin + Jetpack Compose, Media3/ExoPlayer + FFmpeg extension, indigo→violet dark-first design. The project is currently an empty Android scaffold; no source files exist yet beyond placeholder tests.

Full product docs are in `vix-play-docs/media-player-docs/`. Design assets (color tokens, typography, icons, launcher icon) are in `config-vix-play/player-design-assets/`.

## Build commands

```bash
# Assemble debug APK
./gradlew assembleDebug

# Install on connected device
./gradlew installDebug

# Run unit tests
./gradlew test

# Run instrumented tests (device/emulator required)
./gradlew connectedAndroidTest

# Run a single unit test class
./gradlew test --tests "com.devbytes.vixplay.ExampleUnitTest"

# Lint
./gradlew lint

# Clean
./gradlew clean
```

Gradle configuration cache is enabled (`gradle.properties`). If build scripts change unexpectedly, run `./gradlew clean` first.

## Architectural decisions

These were locked in during design review — don't revisit without a specific reason:

| Concern | Decision |
|---|---|
| **Module structure** | Single-module for now. Extract to multi-module after MVP compiles and plays a video. The tech-architecture doc shows the target multi-module layout for reference only. |
| **DI** | Hilt. `PlayerController` scoped `@ActivityRetainedScoped` — survives rotation, torn down on back. Migrate to `MediaSessionService` when background audio lands (Roadmap Step 5). |
| **Navigation** | Navigation Compose with `@Serializable` destination classes (2.8+). `ACTION_VIEW` external intents map to deep-link URIs. |
| **Application ID** | `com.devbytes.vixplayer.app` — this is permanent once the Play Store listing is created. |
| **Storage** | MediaStore + granular media permissions as primary; SAF tree access for folder browsing; `MANAGE_EXTERNAL_STORAGE` as opt-in power mode only (with Play Console declaration). |
| **FFmpeg** | Media3 FFmpeg extension built via Google's provided script, checked in as a local `.aar`. No custom FFmpeg build. |
| **AC3/DTS** | Omitted. HW decode covers most devices. Show the codec name in the unsupported-format error; revisit licensing at 1.0. |
| **Subtitles** | Media3 built-in renderer (SRT + basic SSA) for MVP. libass (full styled ASS/SSA) is a fast-follow once playback is stable — not a blocker. |
| **Pro entitlement** | Local Play verification via Google Play Billing v6. No backend verification until a subscription tier exists. |
| **Analytics** | Firebase Crashlytics only. No Firebase Analytics for MVP — add it at beta when real users exist. `google-services.json` goes in `app/`. |
| **Ads** | AdMob. Banner only in library chrome; never on the player surface. No interstitials during playback. |
| **Theming** | Single `PlayerTheme` composable with an internal `Build.VERSION.SDK_INT >= 31` branch for dynamic color. No hardcoded colors anywhere — every component reads semantic tokens from `Tokens.kt`. |
| **Android TV** | Compose for TV (`androidx.tv`), not Leanback. Keeps the entire UI layer in one paradigm. |
| **Private folder recovery** | Recovery code shown at PIN setup (hashed locally). No email recovery — that requires a backend. |

## Build order

**First milestone:** Steel thread — pick a file from a folder browser → hand URI to Media3 → video plays full-screen with play/pause and seek. Nothing else ships until this works on a real device. This validates Media3 + FFmpeg setup, Navigation Compose routing, and Hilt wiring end-to-end.

Roadmap steps after that (from `product/04-roadmap-mvp.md`):
1. Foundations — `PlayerTheme`, `GestureDispatcher`, Room schema, storage access layer
2. Playable core — FFmpeg extension, folder browser, video player, resume position
3. Subtitles — SRT/ASS via Media3, external load, sync offset
4. Feel differentiators — precision seek + thumbnails, variable-speed hold, haptics
5. Modern shell — Material You, AMOLED, video library home, audio player + `MediaSessionService`
6. Reach — network streaming, Chromecast, PiP, equalizer
7. Polish — gesture remap UI, subtitle online search, playlists, private folder, ambient backglow, TV

## Key constraints

- **No hardcoded colors.** Every component reads semantic tokens. This is the #1 theming rule — retrofitting is the worst rewrite.
- **No ads on the player surface.** AdMob banners only in library chrome.
- **AC3/DTS omitted** — do not add software decoders without legal review of the Dolby/DTS license situation.
- **FFmpeg must be LGPL build** — audit build flags before adding any GPL-only components; GPL taints the whole app.
- `minSdk = 24`, `targetSdk = 36`. Material You (`dynamicColorScheme`) requires runtime API 31+ check; `PlayerTheme` handles this internally.
- Resume positions use MMKV (not DataStore) — DataStore's coroutine overhead is too slow for per-seek writes.

### 2026-07-23 — Subtitle styling (presets + per-property) — DONE, **not yet device-verified**
- **Corrects an earlier claim in this log.** Subtitle styling was repeatedly described as "blocked on libass". **It wasn't.** `javap` on media3-ui 1.6.1 shows `SubtitleView` exposes the PRD's entire styling list natively: `setStyle(CaptionStyleCompat)` (foreground/background/window colour, `edgeType`+`edgeColor`, `typeface`), `setFractionalTextSize`, `setBottomPaddingFraction`, and **`setApplyEmbeddedStyles(boolean)`** — which *is* the PRD's "respect or override embedded ASS styles". libass only governs **fidelity of embedded ASS rendering** (karaoke, complex positioning, effects); it was never a prerequisite for *user* styling. The libass entry in the architecture table still stands for ASS fidelity.
- **PRD:** `features/subtitles.md` → "Style presets (Netflix/Cinema/Minimal/Classic) + font/size/color/outline/box/position with live preview. Respect or override embedded ASS styles."; `screens/07-subtitle-manager.md` lists **Style** as a sheet action; `SubtitleDefaultsScreen` had been a dead stub since the Settings pass.
- **Q1=A — one shared `SubtitleStyleEditor`, hosted in BOTH places.** Full-screen in `SubtitleDefaultsScreen` (**retiring the stub**; where persisted defaults belong, same shape as playback speed) and as a `PlayerSheet.SubtitleStyle` bottom sheet off the subtitle sheet's new `Style` row. Both read/write the same DataStore-backed `SubtitleStyle`, so they cannot disagree — one component, one source of truth, two hosts. `showPreview` is **true** in settings (sample caption strip) and **false** in the player, where the real subtitles on the video are a better preview than any sample. Rejected B (settings-only — leaves the PRD's sheet action absent, or navigating out of the player mid-video) and C (player-only — leaves the settings stub dead and puts a persisted global preference in a transient surface).
- **Q2=A — 4 presets + curated per-property controls.** `SubtitlePreset { NETFLIX, CINEMA, MINIMAL, CLASSIC, CUSTOM }`; picking one replaces every visual property, and touching any control flips the label to **Custom**. Size and position are sliders; edge and font are chip rows. Rejected B (presets only — "make it bigger", the single most-wanted adjustment, becomes unavailable unless a preset happens to match) and C (full RGB picker — net-new component, and unconstrained colour lets users produce genuinely unreadable captions).
- **Two deliberate narrowings of the PRD, logged not hidden:** (1) **"color" is a fixed swatch palette**, not arbitrary RGB — white/yellow/cyan/green for text, transparent→solid black for the box; all stay high-contrast against the dark backgrounds subtitles sit on, so no combination can produce an unreadable result. (2) **"font" is the four system typefaces** (Default/Sans/Serif/Mono), not a font picker — Media3 takes a `Typeface`, and bundling or scanning custom fonts is a separate feature.
- **Q3=A — global DataStore persistence; respect embedded styles by default, with an explicit override toggle.** Style is a stable preference (like playback speed), **not** per-file like the sync offset — desync is a property of one release, appearance is a property of the user. Default `overrideEmbedded = false`: a deliberately-styled fansub renders as authored, and the user's style applies to the SRT files that carry none, which is most of them. Rejected always-override (flattens every intentionally-styled release with no way back) and always-respect (no recourse — exactly the complaint the PRD's "or override" clause anticipates).
- **Deliberate asymmetry: text size always wins.** `applySubtitleStyle` sets `setApplyEmbeddedStyles(!override)` but **`setApplyEmbeddedFontSizes(false)` unconditionally**, so the size slider takes effect even on ASS files carrying their own sizes. Without it, the most-wanted adjustment would silently do nothing on exactly the files people complain about, sending them hunting for a toggle they shouldn't need. Size therefore behaves differently from colour/position — a real trade, taken knowingly.
- **Defaults are Media3's own** (`textSizeFraction = 0.0533f`, `bottomPaddingFraction = 0.08f`, white, outline edge), so an untouched install renders exactly as it did before this change.
- Files: `+data/repository/SubtitleStyle.kt` (`SubtitleStyle`, `SubtitlePreset`, `SubtitleTypeface`, `SubtitleColors`, `SubtitleEdge`); `+ui/player/components/SubtitleStyleEditor.kt` (shared editor + `SubtitleView.applySubtitleStyle`); `+ui/settings/SubtitleDefaultsViewModel.kt`; `SettingsRepository.kt` (8 keys + `subtitleStyle` flow + `setSubtitleStyle`); `SubtitleDefaultsScreen.kt` (stub → real editor); `SubtitleSheet.kt` (+`onStyle` + Style row); `PlayerSheet.kt` (+`SubtitleStyle`); `PlayerViewModel.kt` (+`subtitleStyle`/`setSubtitleStyle`); `PlayerScreen.kt` (collect style, `LaunchedEffect(playerView, subtitleStyle)` → `applySubtitleStyle`, sheet case + row wiring). Compiles clean; `assembleDebug` OK.
- **Deferred:** custom font loading, per-file style overrides, secondary-subtitle styling (P2), full ASS fidelity (libass — still genuinely blocked), and `showSubtitlesByDefault` (still an orphaned `SettingsRepository` key; needs track-selection wiring at prepare time, not a styling concern).

### 2026-07-23 — Audio groupings: Albums · Artists · Folders tabs — DONE, **not yet device-verified**

A fifth tab would be selectable and permanently empty: the exact dead-UI shape refused everywhere else in this project (and the reason these four groupings were deferred rather than stubbed in the previous entry). Playlists keeps the separate route + existing `onPlaylistsClick` it already has, so adding the tab later is one line. Rejected B (all five with a placeholder tab) and C (skip Folders — arbitrary; Folders is the same `groupBy` and is the grouping that *works for untagged libraries*, which is exactly who the fallback paths serve).
- **Q2=A — in-place drill, mirroring `FolderBrowserScreen`.** `openGroup: StateFlow<AudioGroup?>` — null shows the grouping list, non-null shows that group's tracks with its name as the title and a back arrow. `BackHandler(enabled = openGroup != null)` so system back leaves the drill before the screen; `selectTab` clears the drill. Tabs hide while drilled in (title + back own that level). **No new routes or screens**, and the track row/player wiring is reused verbatim. Rejected B (three new routes + screens, when nothing currently links to them — promotable later if search or the player ever needs "go to album") and C (accordion — rows that triple in height wreck scrolling, and it collapses for artists with many albums).
- **Q3=A — all groupings derived in memory from the single `queryAllTracks()`.** `groupBy` on album / artist / folder. Rejected MediaStore's `Audio.Albums`/`Audio.Artists` tables: they are the "official" path **but don't know about our `IS_MUSIC != 0` filter**, so their `NUMBER_OF_TRACKS` could disagree with the tracks actually listed — a real inconsistency for no benefit at this size, plus Folders has no MediaStore table anyway and would need a third code path. In memory, every grouping inherits the filter and the `<unknown>`-artist normalisation by construction, and costs a `groupBy` on data already loaded for the Tracks tab.

DATA is deprecated but still populated; `BUCKET_DISPLAY_NAME` needs API 29 and `minSdk = 24`, and `MediaRepository` already uses DATA for video paths, so this matches existing practice. Blank/missing → "Unknown folder".
- **Context queue generalised.** `play(track, context)` takes the list the track was tapped in, so playing from an album queues **that album**, not the whole library — the PRD's "adds context queue". The Tracks tab passes the full library, a drill passes the group. One rule, no special cases.

### 2026-07-23 — Audio mini-player + MediaMetadata on the queue — DONE, **not yet device-verified**
- **PRD:** `screens/04-audio-library.md` → "Persistent **mini-player** at bottom when audio is active (art, title, play/pause, next)", "Swipe mini-player up → full Audio Player", acceptance "mini-player reflects live playback state across tabs".
- **The trap the audit found:** `PlayerScreen`'s dispose calls `player.stop()`, and **`stop()` keeps the playlist** — it doesn't clear it. So after watching a video and returning to the library, `mediaItemCount > 0` with a **video** item loaded. A naive "show when something is queued" mini-player would pop up advertising the video the user just exited, with fallback art and no title.
- **Q1=A — `PlayerController` tracks `PlaybackKind { NONE, VIDEO, AUDIO }`**, set by `prepareFor` (VIDEO) / `prepareQueue` (AUDIO), cleared by `stop()`, exposed as a `StateFlow`. Mini-player shows when `kind == AUDIO`. One authoritative field on the class that already owns the player, set at exactly the two places media is loaded. Rejected B (resolve the current URI against the audio-library map — inference where a fact is available; needs the whole library loaded to answer a question about the *player*, and fails for audio opened by external intent) and C (`clearMediaItems()` on leaving the video player — throws away the loaded item and makes the two paths asymmetric purely for the mini-player's convenience). **Bonus:** keying on `kind` rather than `isPlaying` makes the **paused** case correct for free — the bar stays up while audio is loaded but paused, instead of vanishing and leaving no way to resume.
- **Q2=A — tap to expand + swipe-up, play/pause + next, thin progress line.** Tap is the discoverable primary (nobody's first instinct is to swipe a bar they just noticed); an upward `detectVerticalDragGestures` past −48f satisfies the PRD's gesture; they don't conflict. 2dp `primary` progress fill along the top edge, polled at 500ms. Rejected swipe-only (undiscoverable as the sole entry; the PRD lists the gesture without saying it's exclusive) and dropping `next` (explicitly in the PRD, and skipping without opening the full player is most of the point). **No swipe-to-dismiss** — not in the PRD, and "dismiss" is ambiguous on a shared player (stop playback, or hide the bar and leave audio running with no visible control?); the notification already provides a stop path.
- **Q3=A — `MediaMetadata` travels on the `MediaItem`; fixes a latent defect in shipped code.** `prepareQueue` now takes `QueueItem(uri, title, artist, album, artworkUri)` instead of bare URI strings and builds items via `setMediaMetadata(...)`. Previously it built bare `MediaItem.fromUri(uri)` — **and the MediaSession notification/lock screen read exactly that**, so the background playback shipped in the previous entry was very likely showing a **notification with no title, artist or artwork**. Now the mini-player reads `player.mediaMetadata` with **no library query at all**, `AudioPlayerViewModel`'s URI→`AudioTrack` map and its full-library query are **deleted**, and the notification gets correct metadata for free. Rejected a shared singleton map (leaves the notification blank; keeps metadata parallel to the player) and a `List<AudioTrack>` inside `PlayerController` (same objection, and it would have to stay index-aligned with a playlist that shuffle reorders). Same principle as the queue decision: the player is the single source of truth.
- **Hosting:** `MiniPlayer` sits **inside the existing `bottomBar` slot**, stacked above `VixBottomNav` in a `Column`, so Scaffold padding accounts for both and no content hides behind it. Already excluded from full-screen routes by the existing `isFullScreen` check (player/audio-player/splash), so it can't appear over the player it expands into. Expanding navigates to `audioPlayerRoute(NOW_PLAYING_ID = -1L)` — a sentinel `ensureQueued` ignores (non-positive ids return early), so it opens now-playing **without disturbing the queue**.
- Files: `+ui/audio/MiniPlayer.kt`, `+ui/audio/MiniPlayerViewModel.kt`; `PlayerController.kt` (+`PlaybackKind`, +`QueueItem`, +`kind` StateFlow, metadata-building `toMediaItem()`); `AudioLibraryViewModel.kt` (+`AudioTrack.toQueueItem()`); `AudioPlayerViewModel.kt` (−map, −library query); `AudioPlayerScreen.kt` (reads `mediaMetadata`, `onMediaMetadataChanged`); `VixPlayNavGraph.kt` (Column in bottomBar + `NOW_PLAYING_ID`). Compiles clean; `assembleDebug` OK.
- **⚠️ Still unrun on hardware, now three features deep** (background playback → audio slice → this). The metadata fix above is exactly the kind of thing only a device run confirms.
- **Deferred:** swipe-to-dismiss, queue view/reorder, seek from the mini-player, marquee for long titles, shared-element transition into the full player.

### 2026-07-23 — Background playback / MediaSessionService (Step 5 keystone) — DONE, **not yet device-verified**
- **⚠️ Compiles and `assembleDebug` passes, but the runtime behaviour (notification, foreground-service promotion, audio focus, task-removal) has NOT been exercised on a device or emulator.** Everything below is a design record, not a verification. First device run should check: notification appears + controls work, audio survives app-swipe with the toggle on, playback pauses on background with it off, PiP still keeps playing.
- **PRD:** `features/background-pip.md` → `MediaSessionService` hosts the player, foreground service type `mediaPlayback`, rich notification + lock-screen + Bluetooth/media buttons, audio focus (pause on transient loss, duck, resume).
- **Supersedes a locked CLAUDE.md decision.** The architecture table said `PlayerController` is `@ActivityRetainedScoped`. Background playback requires the player to **outlive the Activity**, so it is now `@Singleton`. That row is obsolete.
- **Q1=B — one `@Singleton` `ExoPlayer` via Hilt; the service hosts the `MediaSession`; the UI keeps calling the player directly. No `MediaController`.** The app is single-process, so `PlaybackService` and `PlayerScreen` hold the *same* object. The **session** is what publishes to notification/lock-screen/Bluetooth — a **controller** buys only cross-process isolation this app doesn't use, and would cost three concrete regressions: (1) `MediaController` connects **asynchronously** → a null-player window every call site must handle; (2) `Player` (the interface) exposes **no `videoFormat`/`audioFormat`** → the **File Info sheet's Stream section would regress**; (3) the subtitle-offset holder would sit across the session boundary → each offset change needs a **custom session command** instead of a field write. Rejected A (textbook split) for those costs and C (no service) as impossible — surviving app-swipe requires a foreground service.
- **New `player/PlayerController.kt` (`@Singleton`)** owns the `ExoPlayer`, the `SubtitleOffsetHolder`, `setAudioAttributes(…, handleAudioFocus = true)` (this is what satisfies the PRD's audio-focus criterion — pause/duck/resume for free) and `setHandleAudioBecomingNoisy(true)`. **`prepareFor(uri, subtitleOffsetMs)` is the single entry point** for opening a video — the accepted cost of an app-scoped player is that per-file state no longer resets when the composable leaves scope, so it must reset explicitly or a previous video's subtitle offset silently applies to the next. `stop()` frees decoders **without releasing** — a released player would leave the singleton holding a dead object.
- **New `player/PlaybackService.kt` (`MediaSessionService`, `@AndroidEntryPoint`)** builds the session over the injected player. `onTaskRemoved` → if nothing is playing, `controller.stop()` + `stopSelf()`; otherwise keep running so audio continues. `onDestroy` releases **only the session**, never the player.
- **Q2=A — engine only; the audio library/player stay placeholders.** The roadmap bundles "audio player + MediaSessionService", but the audio library is a whole separate data layer (MediaStore *audio* queries, albums/artists, new repo + VM) plus a second player surface — three or four grills of decisions, none of them needed for background playback. Splitting also keeps a service-lifecycle bug and a MediaStore-query bug out of the same increment. The two audio screens are now blocked on **UI work, not a missing engine**.
- **Q3=A — opt-in persisted setting, default OFF; PiP wins.** `SettingsRepository.backgroundPlayback` (DataStore) + a real `Background playback` switch that **replaces `PlaybackDefaultsScreen`'s placeholder** (new `PlaybackDefaultsViewModel`; retires more of the orphaned SettingsRepository, same path playback speed took). Default off because a video player that silently keeps playing after backgrounding reads as a battery bug.
